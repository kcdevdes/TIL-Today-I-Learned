# 스프링 기본

## 객체 지향이란

- 코드를 유연하고 변경이 용이하게 만드는 것
- 레고 블럭 조립하듯이 컴포넌트를 쉽고 유연하게 만든다
- 이를 다형성(Polymorphism)이라고 한다.

## 역할과 구현을 분리

- 역할과 구현으로 구분하면 단순해지고, 우연해지며, 다양해진다.
- 클라이언트는 대상의 역할만 알면 된다.
- 클라이언트는 구현 대상의 내부 구조를 몰라도 된다.
- 클라이언트는 구현 대상의 내부 구조가 변경되어도 영향을 받지 않는다.
- 클라이언트는 구현 대상 자체를 변경해도 영향을 받지 않는다.

## 자바 언어의 다형성

- 역할 = 인터페이스
- 구현 = 인터페이스를 구현한 클래스, 구현 객체
- 객체를 설계할 때 역할과 구현으로 명확히 분리

## 자바 언어의 다형성

- 오버라이딩 된 메서드가 실행
- 다형성으로 인터페이스를 구현한 객체를 실행 시점에서 유연하게 변경할 수 있는 것.

## SOLID 원칙

- Single Responsibility Principle
    - 한 클래스는 하나의 책임만 가진다.
    - 기준은 변경이다. 변경이 있을 때 파급 효과가 적으면 된다.
- Open Closed Principle
    - 확장에는 열려있으나 변경에는 닫혀있는다.
    - 다형성을 활용한다.
    - 확장은 클래스를 하나 만들어서 새로운 기능을 구현한다.
- Liskov Substitution Principle
    - 다형성에서 하위 클래스는 인터페이스 규약을 다 지켜야 한다는 것. 다형성을 지원하기 위한 원칙을 지키라는 것
- Interface Segregation Principle
    - 특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스 하나보다 낫다.
    - 자동차 인터페이스 → 운전 인터페이스, 정비 인터페이스로 분리
    - 사용자 클라이언트 → 운전자 클라이언트, 정비사 클라이언트로 분리
- Dependency Inversion Principle
    - 추상화에 의존해야지, 구체화에 의존하여선 안된다.
    - Role에 의존하게 만들라.
